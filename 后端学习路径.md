## 第一阶段：地基（2-3 周）

### 你不需要重学 Python，但需要补这些

| 知识点 | 学到什么程度 | 为什么 |
|:---|------|------|
| **虚拟环境** (venv/conda) | 每个项目独立环境，`pip freeze > requirements.txt` | 项目不打架 |
| **类型注解** | `def f(x: int) -> str:` 能读写 | FastAPI 靠类型注解工作 |
| **装饰器** | 会写 `@app.get()` 就行了，不用会写装饰器 | FastAPI 核心语法 |
| **上下文管理器** | `with open() as f:` 和 `async with` | 数据库连接、文件操作都在用 |
| **列表/字典推导式** | `[x for x in data if x > 0]` | 数据处理写到飞起 |
| **异常处理** | `try/except/finally` | 服务不能一报错就挂 |

### Git：不学没法干活

```bash
# 你就需要这 8 个命令，别的用到再查
git init                          # 初始化仓库
git clone <url>                   # 克隆项目
git add <file>                    # 暂存修改
git commit -m "做了什么"           # 提交
git push                          # 推到远程
git pull                          # 拉取最新
git branch <name>                 # 新建分支
git checkout -b <name>            # 切换并新建分支
```

**理解**：GitHub 是远程仓库，本地写代码 commit 后 push 上去，队友 pull 下来。分支让你在不影响主代码的情况下开发新功能。

### Linux 命令行：服务器上没有图形界面

```
必须会的：
  ls cd pwd                     ← 看文件、进目录、我在哪
  mkdir rm cp mv                ← 创建、删除、复制、移动
  cat head tail grep            ← 查看文件内容、搜索
  ps aux | grep python          ← 查进程
  kill -9 <pid>                 ← 杀进程
  chmod +x                      ← 加执行权限
  curl http://localhost:8000    ← 测试 API（相当于浏览器的地址栏）
```

### HTTP 深入

除了 Day 4 笔记的基础，补充：

| 知识点 | 要理解到什么程度 |
|:---|------|
| **请求头常见字段** | `Content-Type`, `Authorization`, `Accept`, `User-Agent` |
| **Cookie vs Token** | Cookie 是浏览器自动带的，Token 是代码手动加的 |
| **HTTPS 原理** | SSL/TLS 握手（面试会问），平时知道就行 |
| **跨域 CORS** | 前端 `localhost:3000` 调后端 `localhost:8000` 被浏览器拦截 → 后端加 CORS 头 |
| **WebSocket** | HTTP 只能客户端先说话；WebSocket 服务器也能主动推消息 |

**检验标准**：打开浏览器 F12 → Network 标签，随便点一个请求，能看懂 Request Headers、Response Headers、Preview 里的 JSON。

## 第二阶段：框架 + 数据库（3-4 周）

### FastAPI 掌握清单

| 阶段 | 内容 | 检验方式 |
|:---|------|------|
| 入门 | 路由、路径参数、查询参数、请求体 | 能写 4 个 CRUD 接口 |
| 进阶 | 依赖注入 `Depends()`、中间件、异常处理器 | 能写一个认证依赖 |
| 进阶 | BackgroundTasks 后台任务 | 知道发邮件/数据处理不阻塞响应 |
| 进阶 | WebSocket 端点 | 能写一个简单的聊天端点 |
| 进阶 | `lifespan` 启动/关闭事件 | 服务启动时连接数据库，关闭时断开 |

### SQL 掌握清单

数据库分"会用"和"会设计"，两个都要过关。

**会用 SQL**：
```sql
-- 这些全部能手写，不看文档
SELECT ... FROM ... WHERE ... ORDER BY ... LIMIT ...
SELECT ... FROM a JOIN b ON a.id = b.a_id          -- 联表
SELECT ... GROUP BY ... HAVING ...                  -- 分组统计
INSERT INTO ... VALUES ...
UPDATE ... SET ... WHERE ...
DELETE FROM ... WHERE ...
CREATE TABLE ... (id PRIMARY KEY, ..., FOREIGN KEY ...)
CREATE INDEX idx_name ON table(column)              -- 索引
```

**会设计表结构**（更重要）：
- **范式化**：每个表只存一种东西，不冗余（用户表和订单表分开）
- **一对多**：订单里存 `user_id` 外键
- **多对多**：用户-角色关系需要一张中间表 `user_roles(user_id, role_id)`
- **索引**：给常查询的列加索引（WHERE 后面的列），但别乱加（写入变慢）

### Redis 掌握清单

| 场景 | 用什么命令 | 实际例子 |
|:---|------|------|
| 缓存 | `SETEX key 秒 value` | 缓存天气结果 10 分钟 |
| 限流 | `INCR key` + `EXPIRE key 60` | 每分钟最多 20 次调用 |
| 分布式锁 | `SET key value NX EX 10` | 同一时间只有一个 agent 处理某任务 |
| 会话 | `SETEX session:token 3600 user_json` | 用户登录状态存 1 小时 |
| 排行榜 | `ZADD key score member` | agent 使用次数排行 |

### 认证与授权（JWT）

这是面试必问 + 实际必用的。

```
用户登录流程：
1. 用户发 POST /login {"username":"alice","password":"xxx"}
2. 服务器验证密码，生成 JWT Token（一串加密字符串，内含 user_id + 过期时间）
3. 用户以后每次请求都在 Header 里带 Authorization: Bearer <token>
4. 服务器在 Depends 里解密 token，拿到 user_id，判断权限

JWT 三部分（用 . 分隔）：
  Header.Payload.Signature
  └─算法┘ └─数据┘ └─签名防篡改┘
```

**关键认知**：JWT 本身不加密（Payload 是 base64，任何人能解码看到内容），它只防篡改（一改就验签失败）。不要把密码放 JWT 里。

### Python ORM（SQLAlchemy）

写裸 SQL 可以，但生产代码用 ORM 更多——把数据库表映射成 Python 类：

```python
# 原生 SQL
cursor.execute("SELECT * FROM users WHERE id = ?", (5,))

# ORM 方式（SQLAlchemy）
user = session.query(User).filter(User.id == 5).first()
print(user.name)
```

选学：知道有 ORM 这东西就行，入门阶段写原生 SQL 更重要。

## 第三阶段：中间件 + 部署（3-4 周）

### Docker 容器化

**为什么必须学**：你的代码在自己电脑上能跑，到服务器上就挂——"我本地没问题啊"。Docker 把代码 + 运行环境打包成一个镜像，到哪都一样。

```dockerfile
# Dockerfile 示例 —— 把你的 FastAPI 服务容器化
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["uvicorn", "agent_server:app", "--host", "0.0.0.0", "--port", "8000"]
```

```bash
docker build -t my-agent .          # 构建镜像
docker run -p 8000:8000 my-agent    # 运行容器
```

**必须会的 Docker 命令**：`build`, `run`, `ps`, `stop`, `logs`, `exec -it`, `compose up/down`

**Docker Compose**（同时管理多个容器）：
```yaml
# docker-compose.yml —— 一键启动 agent + Redis + 数据库
services:
  app:
    build: .
    ports: ["8000:8000"]
    depends_on: [redis, db]
  redis:
    image: redis:7
    ports: ["6379:6379"]
  db:
    image: postgres:16
    environment:
      POSTGRES_DB: agent_db
```

### 消息队列（选学，面试加分）

场景：用户请求 agent 生成一份报告，报告生成要 30 秒，不能让他一直等着。

```
不用消息队列：                    用消息队列：
用户 → 服务器 → 等 30 秒 → 返回    用户 → 服务器 → 立即返回"正在处理"
                                               ↘ 消息队列 → Worker → 生成报告
                                                                     ↓
                                                           WebSocket通知用户
```

**主流选择**：RabbitMQ（老牌稳妥）或 Redis 的 List 也能当简易队列用。

### 日志、监控、告警

| 层级 | 工具 | 干什么 |
|:---|------|------|
| 日志 | Python `logging` 模块 | 记录谁在什么时候调了什么接口 |
| 结构化日志 | `structlog` 或 JSON 格式日志 | 方便 ELK 检索 |
| 监控 | Prometheus + Grafana | 看 QPS、响应时间、错误率、CPU/内存 |
| 告警 | Grafana Alert / 企业微信通知 | 错误率飙升 → 发消息通知你 |

最小可用版：`logging.info(f"{user} called {endpoint} → {status}")`，把日志写到文件里就够了，别上来就搞 ELK。

### CI/CD 自动部署

**CI（持续集成）**：每次 git push，GitHub Actions 自动跑测试。测试不通过不允许合并。
**CD（持续部署）**：测试通过后，自动构建 Docker 镜像并部署到服务器。

```yaml
# .github/workflows/test.yml
name: Run Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements.txt
      - run: pytest
```

## 第四阶段：工程化 + 面试（4-5 周）

### 设计模式（后端常用）

不是所有 23 种都要学。后端最常用的就这些：

| 模式 | 干什么 | 在哪见到 |
|:---|------|------|
| **单例** | 全局只有一个实例 | 数据库连接池、配置对象 |
| **工厂** | 根据参数创建不同对象 | 多种支付方式、多种通知渠道 |
| **观察者** | 事件发生后通知订阅者 | 消息队列、Event Bus |
| **策略** | 算法可替换 | 不同计费规则、不同排序算法 |
| **装饰器** | 不修改原函数加功能 | FastAPI 的 `@app.get()`、日志装饰器 |
| **依赖注入** | 需要的依赖从外部传入 | FastAPI 的 `Depends()` |

### 系统设计（面试大头）

中小公司面试会有 1-2 道系统设计题。套路固定：

**通用思维框架**：
1. **需求澄清**：问清楚要什么功能、多少用户、多少数据
2. **估算容量**：日活、QPS、存储量（随口估算就够）
3. **API 设计**：先定有哪些接口
4. **数据库设计**：画 ER 图（有哪些表、表之间怎么关联）
5. **架构图**：画方框——客户端 → 负载均衡 → 应用服务器 → 数据库 + 缓存
6. **深入细节**：缓存策略、读写分离、消息队列、CDN

**常见题目**（每道都动手画一次）：
- 设计一个短链接系统（TinyURL）
- 设计一个 API 限流系统 ← 和 agent 直接相关
- 设计一个聊天系统
- 设计一个通知推送系统
- 设计一个文件上传服务

### 项目实践（简历上写什么）

至少做 2 个完整项目：

**项目 1：Agent 管理平台**（2 周）
```
功能：
- 用户注册/登录（JWT 认证）
- 创建 Agent（配 system prompt + 选工具）
- 与 Agent 对话（对话历史存数据库）
- 查看对话统计（调用次数、工具使用频率）
- API 限流（Redis）
技术栈：FastAPI + SQLite + Redis + Docker + JWT
```

**项目 2：RAG 知识库问答系统**（2 周）
```
功能：
- 上传文档（PDF/Word/TXT）
- 文档切片 + 向量化 + 存入向量数据库
- 用户提问 → 检索相关文档片段 → 拼进 prompt → LLM 回答
- 回答溯源（标注答案来自哪个文档的哪一段）
技术栈：FastAPI + PostgreSQL + pgvector + OpenAI Embedding
```

两个项目写完，完整部署到服务器（买一台云服务器学生价几十块一个月），放到简历上。

### 八股文（面试要背的）

面试会问的理论基础：

| 分类 | 高频问题 |
|:---|------|
| **网络** | TCP 三次握手四次挥手、HTTP1/2/3 区别、HTTPS 加密过程、DNS 解析过程 |
| **操作系统** | 进程 vs 线程 vs 协程、死锁四个条件、内存管理 |
| **数据库** | 索引数据结构(B+树)、事务 ACID、隔离级别、MVCC |
| **Redis** | 缓存穿透/击穿/雪崩、持久化 RDB vs AOF、淘汰策略 |
| **Python** | GIL、装饰器原理、生成器、垃圾回收、`__init__` vs `__new__` |
| **设计模式** | 单例的各种写法、工厂 vs 抽象工厂 |

## 最重要的建议

1. **每学一个知识点就写代码**，不要连着看三天视频。看完了不写 = 没看。
2. **GitHub 绿点不能断**，每天至少 commit 一次，半年后你的 GitHub 就是最好的简历。
3. **不要追求完美理解**，后端东西太多，先跑通再深究。Docker 第一天就 `docker run hello-world`，别先从 namespace/cgroup 原理看起。
4. **项目驱动**，不是为了学而学，是为了"我要做一个能上线给人用的 agent 平台"而学。
