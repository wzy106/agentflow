## 第一章：什么是后端开发

### 1.1 生活类比

你去餐厅吃饭：

```
你（用户）              服务员（前端）          厨师（后端）
   │                      │                    │
   │ "我要一份蛋炒饭"       │                    │
   ├─────────────────────→│                    │
   │                      │ "蛋炒饭一份！"       │
   │                      ├───────────────────→│
   │                      │                    │ 炒菜...
   │                      │     "菜好了"        │
   │                      │←───────────────────┤
   │    "您的蛋炒饭"        │                    │
   │←─────────────────────┤                    │
```

- **前端（服务员）**：你直接接触的——菜单好看吗、态度好吗
- **后端（厨师）**：你看不到的——菜怎么做、火候多大

你写的 `server.py` 就是厨师。浏览器里那个网页就是服务员。

### 1.2 专业定义

**后端开发**：运行在服务器上，负责处理业务逻辑、操作数据库、调用外部 API、返回数据给前端的程序。

**服务器**：一台一直开着、等待别人来访问的电脑。

### 1.3 你的代码里

```python
# server.py 第 148 行附近
@app.post("/chat")                            # ← 这就是后端接口
async def chat_endpoint(request: Request):    # ← 收到请求
    user_input = body.get("message", "")      # ← 取出用户问题
    # ... 调 DeepSeek、执行工具 ...             # ← 处理业务逻辑
    return StreamingResponse(...)              # ← 返回结果
```

### 1.4 面试会怎么问

> **Q：前端和后端的区别是什么？**
>
> 答：前端运行在浏览器，负责界面展示和用户交互。后端运行在服务器，负责数据处理、业务逻辑和数据库操作。前端通过 HTTP 请求调用后端 API 获取数据。

## 第二章：HTTP 协议

### 2.1 生活类比

HTTP 就是**浏览器和服务器之间的"寄信规则"**。

每封信必须按固定格式写：

```
信封上写：
  GET /chat                    ← 你要干嘛 + 去哪
  Host: localhost:8000         ← 寄给谁
  Content-Type: application/json  ← 信封里装的是什么格式

信封里装：
  {"message": "你好"}           ← 实际内容
```

### 2.2 专业定义

**HTTP（超文本传输协议）**：浏览器和服务器之间传输数据的协议。基于请求-响应模型——客户端发请求，服务器回响应。

### 2.3 常见 HTTP 方法

| 方法 | 含义 | 你的项目里 |
|------|------|-----------|
| `GET` | 我要拿东西 | `@app.get("/")` 返回首页 |
| `POST` | 我要发东西给你 | `@app.post("/chat")` 发送消息 |
| `PUT` | 我要改东西 | 更新用户资料 |
| `DELETE` | 我要删东西 | 删除一条记录 |

`GET` 和 `POST` 的区别：
- `GET`：数据附在网址后面（`/search?q=关键字`），适合查东西
- `POST`：数据放在请求体里（`{"message": "..."}`），适合发东西，数据更长更安全

### 2.4 HTTP 状态码

服务器返回时，会带一个数字告诉你结果：

| 状态码 | 含义 | 什么时候 |
|--------|------|---------|
| `200` | 成功 | 一切正常 |
| `301` | 永久搬家 | 网址换了 |
| `404` | 找不到 | 你访问的路径不存在 |
| `500` | 服务器炸了 | 代码出 bug 了 |

### 2.5 你的代码里

```python
# 浏览器发的请求（你看不到，但你 server.py 收的就是这个）：
POST /chat HTTP/1.1
Host: localhost:8000
Content-Type: application/json

{"message": "1+1等于几"}

# 服务器返回的响应：
HTTP/1.1 200 OK
Content-Type: text/event-stream

data: {"type":"final","content":"1+1等于2"}
```

### 2.6 面试会怎么问

> **Q：GET 和 POST 有什么区别？**
>
> 答：GET 用于获取数据，参数在 URL 里，长度有限，可被缓存。POST 用于提交数据，参数在请求体里，长度不受限，不会被缓存。GET 是幂等的（多次请求结果相同），POST 不是。

## 第三章：服务器与端口

### 3.1 生活类比

你住的小区（你的电脑）有很多房间（端口）：

```
你的电脑 (localhost)
├── 8000号房 → 你的 Agent
├── 8080号房 → 另一个人的服务
├── 3000号房 → 前端开发服务器
└── 3306号房 → MySQL 数据库
```

每个服务占用一个端口号，互不干扰。

### 3.2 专业定义

**服务器**：提供计算服务的设备或程序。可以是物理机、虚拟机、或你电脑上的一个进程。

**端口**：16 位整数（0-65535），用于区分同一台机器上的不同网络服务。0-1023 是系统保留端口。

### 3.3 常见端口号

| 端口 | 服务 |
|------|------|
| 80 | HTTP 网页 |
| 443 | HTTPS 加密网页 |
| 8000 | Python 开发服务器常用 |
| 3306 | MySQL 数据库 |
| 6379 | Redis 缓存 |

### 3.4 你的代码里

```python
# server.py 最后几行
uvicorn.run(app, host="0.0.0.0", port=8000)
#              │               └── 端口号
#              └── 监听所有网卡（局域网内其他设备也能访问）
```

`0.0.0.0` 和 `127.0.0.1` 的区别：
- `127.0.0.1`（即 localhost）：只有本机能访问
- `0.0.0.0`：局域网内其他设备也能通过你的 IP 访问

### 3.5 面试会怎么问

> **Q：如果端口被占用怎么办？**
>
> 答：换一个端口号。或者查是谁占用的：Windows 用 `netstat -ano | findstr :8000`，找到 PID 后 kill 掉。

## 第四章：路由

### 4.1 生活类比

路由 = 公司的前台。

```
访客：我要找 HR 部门 → 前台指路 → HR 办公室（/hr）
访客：我要找技术部   → 前台指路 → 技术部办公室（/tech）
访客：我要投诉       → 前台指路 → 客服办公室（/complaint）
```

每个部门（函数）对应一个门牌（路径）。前台（路由系统）负责把人引到正确的地方。

### 4.2 专业定义

**路由（Routing）**：把 HTTP 请求的路径和方法映射到对应的处理函数。

### 4.3 你的代码里

```python
# 三个路由，三个路径，三个功能

@app.get("/")                    # 路径：/
async def home():                # 触发：访问首页
    return "<html>...</html>"

@app.post("/chat")               # 路径：/chat
async def chat_endpoint(request): # 触发：发送消息
    ...

@app.post("/rag/upload")         # 路径：/rag/upload
async def rag_upload(request):   # 触发：上传文档
    ...
```

### 4.4 RESTful 风格（加分概念）

业界推荐的路由命名风格：

```
GET    /users          → 获取用户列表
GET    /users/123      → 获取 ID 为 123 的用户
POST   /users          → 创建新用户
PUT    /users/123      → 修改 ID 为 123 的用户
DELETE /users/123      → 删除 ID 为 123 的用户
```

同一个路径 `/users`，不同 HTTP 方法干不同的事。

### 4.5 面试会怎么问

> **Q：什么是 RESTful API？**
>
> 答：一种 API 设计风格。用 HTTP 方法（GET/POST/PUT/DELETE）表示操作类型，用 URL 路径表示资源。同一个路径 `/users`，GET 获取，POST 创建，PUT 修改，DELETE 删除。

## 第五章：API 接口

### 5.1 生活类比

API = 餐厅的**菜单**。

你不需要知道厨房怎么做菜（内部实现），你只需要看菜单（API 文档）：
- 菜名是什么（路径）
- 需要什么食材（参数）
- 做出来是什么（返回值）

### 5.2 专业定义

**API（应用程序接口）**：两个程序之间通信的约定。规定"发什么格式的数据过来，我返回什么格式的数据给你"。

### 5.3 你的代码里

你的 Agent 提供了一个 `/chat` API：

```
接口文档：

  路径：POST /chat
  请求格式：{"message": "你的问题"}
  返回格式：SSE 流，包含以下事件：
    - {"type": "tool", "tool": "calculate", "result": "..."}
    - {"type": "final", "content": "最终回答"}
  
  示例：
    请求：{"message": "1+1等于几"}
    返回：{"type": "final", "content": "1+1等于2"}
```

### 5.4 另一个例子

天气预报 API：

```
  路径：GET https://api.weather.com/current?city=Beijing
  返回：{"city": "Beijing", "temp": 25, "humidity": 60}
```

你不用管气象局怎么测温度，你只需要知道：传城市名 → 拿温度数据。

### 5.5 面试会怎么问

> **Q：你设计 API 时会考虑什么？**
>
> 答：1) 路径命名清晰（`/chat` 而不是 `/xyz`）；2) 请求和返回都用 JSON；3) 返回合适的 HTTP 状态码；4) 错误时有明确的错误信息；5) 做好参数校验。

## 第六章：JSON 数据格式

### 6.1 生活类比

JSON = **全世界的通用表格**。

```
姓名：张三           →  {"name": "张三", "age": 20, "city": "福州"}
年龄：20
城市：福州
```

不管你用什么语言（Python/Java/JavaScript/C++），大家都认这个格式。

### 6.2 专业定义

**JSON（JavaScript Object Notation）**：一种轻量级数据交换格式。纯文本、语言无关、人类可读。是目前 Web API 最主流的数据格式。

### 6.3 JSON 语法规则

```json
{
  "name": "张三",           // 字符串必须用双引号
  "age": 20,                // 数字不用引号
  "is_student": true,       // 布尔值小写 true/false
  "scores": [85, 90, 78],   // 数组用方括号
  "address": {              // 嵌套对象
    "city": "福州",
    "street": "大学城"
  },
  "phone": null             // 空值用 null（不是 None!）
}
```

### 6.4 你的代码里（两个核心操作）

```python
import json

# 操作一：字符串 → 字典（收到 JSON 时用）
json_str = '{"message": "你好"}'          # ← 这是字符串
data = json.loads(json_str)               # ← 变成字典
print(data["message"])                    # → "你好"

# 操作二：字典 → 字符串（发送 JSON 时用）
data = {"result": "1+1=2"}                # ← 这是字典  
json_str = json.dumps(data)               # ← 变成字符串
print(json_str)                           # → '{"result": "1+1=2"}'
```

### 6.5 常见错误

```python
# ❌ JSON 里不能用单引号
'{"name": "张三"}'        # 正确：外面单引号，里面双引号
"{'name': '张三'}"        # 错误！JSON 不认识单引号

# ❌ JSON 里 Python 的 None 是 null
{"value": None}          # 错误！应该用 null
{"value": null}          # 正确

# ❌ 不能直接对字符串用方括号
s = '{"name": "张三"}'
print(s["name"])          # 报错！s 是字符串不是字典
print(json.loads(s)["name"])  # 正确
```

### 6.6 面试会怎么问

> **Q：JSON 和 Python 字典有什么区别？**
>
> 答：JSON 是字符串格式，可以跨语言传输、存文件。Python 字典是内存里的数据结构，只能在 Python 代码里用。`json.loads()` 把 JSON 字符串转成字典，`json.dumps()` 反过来。

## 第七章：SSE 流式推送

### 7.1 生活类比

```
普通 HTTP = 快递          SSE = 流水线
  │                          │
  你下单                    你站在流水线旁
  等 3 天                   每个零件好了就给你
  一次性收到所有货            一个一个给
  │                          │
```

### 7.2 专业定义

**SSE（Server-Sent Events）**：一种服务器向客户端单向推送数据的技术。基于 HTTP 协议，使用 `text/event-stream` 内容类型。客户端通过 `EventSource` API 接收。

### 7.3 SSE vs WebSocket vs 普通 HTTP

| | 普通 HTTP | SSE | WebSocket |
|------|----------|-----|-----------|
| 方向 | 客户端→服务器→客户端 | 服务器→客户端 | 双向 |
| 连接 | 一次请求一次响应 | 长连接 | 长连接 |
| 浏览器支持 | 原生 | EventSource API | WebSocket API |
| 适用场景 | 查数据 | AI 流式输出、股票推送 | 聊天、游戏 |

### 7.4 你的代码里

```python
# server.py 里的流式推送

async def stream():                          # 异步生成器
    yield "data: {}\n\n".format(              # ← yield = 推一条
        json.dumps({"type": "tool", "tool": "calculate"})
    )
    # 前端立刻看到 "调用了 calculate"
    
    # ... 执行工具 ...
    
    yield "data: {}\n\n".format(              # ← 再推一条
        json.dumps({"type": "final", "content": "答案是 95"})
    )
    # 前端显示最终答案

return StreamingResponse(stream(), media_type="text/event-stream")
#      ↑ 告诉浏览器：这是 SSE 流，别断
```

### 7.5 SSE 消息格式

```
data: {"type": "tool", "tool": "calculate"}\n\n
      └─ JSON 内容 ──┘                        └─ 双换行分隔

data: {"type": "final", "content": "答案是 95"}\n\n
```

- `data:` 前缀 → 浏览器识别
- `\n\n` 双换行 → 消息结束标记

### 7.6 面试会怎么问

> **Q：为什么要用 SSE 而不是普通 HTTP 返回？**
>
> 答：AI 生成回复可能需要 5-10 秒。SSE 让服务器边生成边推送，用户实时看到进展，体验好。同时 SSE 比 WebSocket 更轻量，基于标准 HTTP，浏览器原生支持自动重连。

## 第八章：请求与响应

### 8.1 专业定义

**请求（Request）**：客户端发给服务器的数据包，包含方法、路径、头部、体。

**响应（Response）**：服务器返回给客户端的数据包，包含状态码、头部、体。

### 8.2 请求的结构

```
POST /chat HTTP/1.1              ← 请求行：方法 + 路径 + 协议版本
Host: localhost:8000              ← 请求头：额外的描述信息
Content-Type: application/json    ← 请求头：告诉服务器"我发的是 JSON"
                                 ← 空行分隔
{"message": "你好"}               ← 请求体：实际数据
```

### 8.3 响应的结构

```
HTTP/1.1 200 OK                   ← 状态行：协议 + 状态码 + 描述
Content-Type: text/event-stream   ← 响应头：告诉客户端"我发的是 SSE 流"
                                 ← 空行分隔
data: {"type": "final", ...}      ← 响应体：实际数据
```

### 8.4 你的代码里

```python
@app.post("/chat")
async def chat_endpoint(request: Request):   # request = 整个请求对象
    body = await request.json()             # .json() 解析请求体
    user_input = body.get("message", "")    # .get() 安全取值
    
    # ... 处理逻辑 ...
    
    return StreamingResponse(...)            # 返回响应
```

### 8.5 面试会怎么问

> **Q：一个 HTTP 请求包含哪几部分？**
>
> 答：请求行（方法+路径+协议），请求头（Host、Content-Type 等），请求体（数据）。响应类似：状态行（状态码），响应头，响应体。

## 第九章：async/await 异步编程

### 9.1 生活类比

```
同步（没有 async）：              异步（有 async）：
  你烧水 → 等 5 分钟 → 煮面       你烧水 → 不等 → 切菜
  全程盯着水壶，什么也不干            水开了叫你 → 回来煮面
  总耗时：5分+煮面时间              总耗时：等于最长的那个步骤
```

### 9.2 专业定义

**异步编程**：程序在等待某个耗时操作（网络请求、文件读写）时，不阻塞主线程，而是去处理其他任务。操作完成后通过回调或事件通知程序继续。

### 9.3 两个关键关键字

```python
async def 异步函数():      # async def = 定义一个可以"暂停"的函数
    await 耗时操作()       # await = "这里要等，我先去干别的，好了叫我"

# 普通函数：从头跑到尾，中间不能停
def normal():
    a = 1
    time.sleep(5)    # 睡 5 秒，啥也不干
    return a + 1

# 异步函数：中间可以"暂停"去跑别的
async def async_func():
    a = 1
    await asyncio.sleep(5)    # 暂停！CPU 去处理别的请求
    return a + 1
```

### 9.4 你的代码里

```python
# agent.py 里调 LLM
response = await client.chat.completions.create(...)
#          ↑ LLM 回复可能需要 5 秒
#          await 让出 CPU，处理其他用户的消息

# server.py 里读请求
body = await request.json()
#      ↑ 网络传输需要时间
#      await 让出 CPU
```

### 9.5 什么时候用 async

| 用 async | 不用 async |
|----------|-----------|
| 网络请求（调 API） | 纯计算（for 循环加法） |
| 文件读写 | 内存操作（修改变量） |
| 数据库查询 | 数学运算 |
| 多用户并发 | 单用户脚本 |

### 9.6 面试会怎么问

> **Q：async/await 解决了什么问题？**
>
> 答：解决了 I/O 密集型任务的并发问题。当程序等待网络或磁盘时，不让 CPU 闲着，而是去处理其他任务。比多线程更轻量，不存在线程切换开销和数据竞争问题。

## 第十章：装饰器 `@` 语法

### 10.1 生活类比

装饰器 = 给函数**贴标签**，标签上写着"这个函数是用来干嘛的"。

```python
@app.get("/")           ← 标签：这个函数负责首页
async def home():
    ...

@app.post("/chat")      ← 标签：这个函数负责聊天
async def chat():
    ...
```

框架看到标签，就知道"访问 / 时调这个，访问 /chat 时调那个"。

### 10.2 专业定义

**装饰器**：一种修改函数行为的语法糖。本质是一个接受函数作为参数、返回新函数的高阶函数。

### 10.3 等价写法

```python
# 用 @ 语法（你写的）
@app.get("/")
async def home():
    return "首页"

# 等价于（你不需要写，但理解原理用）
async def home():
    return "首页"
home = app.get("/")(home)   # 把 home 函数注册到路径 "/"
```

### 10.4 面试会怎么问

> **Q：Python 装饰器是什么？**
>
> 答：装饰器是一种在函数定义时修改其行为的机制。本质是一个函数，接收另一个函数作为参数并返回新函数。常用于日志记录、权限检查、路由注册等场景。用 `@` 语法写在被装饰函数上方。

| GET | 我要看页面 | `@app.get("/")` |
| POST | 我要发数据 | `@app.post("/chat")` |
| 路由 | 路径 → 函数的映射 | 所有 `@app.xxx("路径")` |
| API | 程序的"菜单" | `/chat` 接口 |
| JSON | 跨语言的数据格式 | `json.loads()` 和 `json.dumps()` |
| SSE | 服务端边跑边推 | `yield "data: ..."` |
| 端口 | 服务的门牌号 | `port=8000` |
| 请求 | 浏览器发给你的 | `request: Request` |
| 响应 | 你发回给浏览器的 | `return StreamingResponse(...)` |
| async/await | 等的时候去干别的 | `await client.chat.completions.create(...)` |
| 装饰器 | 函数的标签 | 所有 `@` 开头的那行 |
